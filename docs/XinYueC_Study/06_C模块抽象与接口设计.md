# 06. C 模块抽象与接口设计

你现在卡住的点，本质上是：C 没有 class，但大型项目仍然需要“分层和抽象”。这一份给你一个可照搬的做法。

## 1. 模块抽象的目标

一个 C 模块应该至少做到三件事：

- 对外暴露稳定接口，不泄漏内部细节
- 把“会变化的实现”封装在 `.c`
- 让调用方只关心“做什么”，不用关心“怎么做”

## 2. 推荐目录和文件角色

以一个 `vector` 模块为例：

- `vector.h`：对外 API、类型声明、配置项
- `vector.c`：公共 API 的实现和参数校验
- `vector_internal.h`（可选）：仅模块内部共享的结构和宏
- `vector_virtual.c`（可选）：复杂逻辑拆分文件

这和 XinYueC 的 `XVector.h + XVector.c + XVector_virtual.c` 基本一致。

## 3. 先定义“接口层”再写实现

先写头文件接口，不要一上来写实现。接口建议分四组：

1. 生命周期：`create/init/deinit/delete`
2. 核心操作：`push/pop/at/remove`
3. 状态查询：`size/capacity/isEmpty`
4. 可配置行为：`setCompare/setCopy/setDeinit`

这样你后面加功能时，不会把 API 写成“函数堆”。

## 4. 把参数校验放在统一入口

XinYueC 的做法很典型：`XVector.c` 做统一校验和转发，实际行为在 `XVector_virtual.c`。

你也可以用同样模式：

```c
bool Vector_push_back(Vector* v, const void* value)
{
    if (v == NULL || value == NULL) return false;
    return VVector_push_back_impl(v, value);
}
```

好处是：调用层错误统一处理，实现层更聚焦算法。

## 5. 用“行为注入”替代写死逻辑

C 模块最常见的抽象手段是函数指针回调，而不是继承。

例如容器可注入：

- `compare`
- `copy`
- `move`
- `deinit`

这正是 `XContainerObject` 的核心思想：把“元素行为”从容器代码里解耦。

## 6. 接口设计的三个硬规则

### 规则 1：名字必须表达语义

像 XinYueC 的 `*_base`、`*_move_base`，虽然偏长，但语义明确。  
你的接口命名也要至少区分：

- copy vs move
- init vs create
- deinit vs delete

### 规则 2：所有权必须写清楚

每个函数都要明确：

- 入参是否会被持有
- 返回值是否需要释放
- 失败后对象是否保持可用

### 规则 3：错误路径必须一致

统一返回风格：

- 查询类接口返回值
- 修改类接口返回 `bool`
- 失败不污染对象状态

## 7. 一个可复用的模块骨架

```c
// xxx.h
typedef struct Xxx Xxx;

Xxx* Xxx_create(size_t typeSize);
void Xxx_init(Xxx* obj, size_t typeSize);
void Xxx_deinit(Xxx* obj);
void Xxx_delete(Xxx* obj);

bool Xxx_push_back(Xxx* obj, const void* value);
void* Xxx_at(Xxx* obj, size_t index);
size_t Xxx_size(const Xxx* obj);
bool Xxx_isEmpty(const Xxx* obj);
```

```c
// xxx.c
Xxx* Xxx_create(size_t typeSize)
{
    if (typeSize == 0) return NULL;
    Xxx* obj = XMalloc(sizeof(Xxx));
    if (obj == NULL) return NULL;
    Xxx_init(obj, typeSize);
    return obj;
}
```

先把这种骨架写顺，再谈复杂抽象。
