# 05. 当前工程中的接入与注意事项

这一份是结合你现在这个 STM32 工程整理的。

## 1. 当前工程里已经怎么接进来了

你现在的工程里，XinYueC 已经至少接到了这些位置：

- `XinYueC/` 下源码已经加入工程
- `XinYueC/include/` 已加入头文件路径
- `IO/XinYue/test_vector.h` 和 `IO/XinYue/test_quickSort.h` 已作为学习测试入口
- `USER/main.c` 已经直接调用了 `test_xvector()` 和 `test_arr_quicksort()`

也就是说，你后续最好的学习方式不是另起一个工程，而是继续在当前工程里加小测试。

## 2. 推荐的测试写法

建议继续沿用你现在这个风格：

1. 在 `IO/XinYue/` 新建一个 `test_xxx.h`
2. 在里面写一个 `void test_xxx(void)` 函数
3. 在 `USER/main.c` 里临时调用
4. 用串口打印结果验证

这样最适合单片机环境，不依赖 PC 端测试框架。

## 3. 比较函数是很多接口的前提

这一点非常关键。

### 原生数组排序

这类接口会直接传入比较函数：

```c
XQuickSort(arr, n, sizeof(int), (XCompare)XCompare_int, XSORT_ASC);
```

### 容器排序

这类接口依赖容器内部的比较函数：

```c
XContainerSetCompare(vec, (XCompare)XCompare_int);
XVector_sort_base(vec, XSORT_ASC);
```

所以你以后只要看到：

- `sort`
- `find`
- `priority queue`

第一反应就应该是：这个类型的 `compare` 配好了没有。

## 4. 如果元素里有动态内存，不能只靠 `memcpy`

比如结构体成员里有：

- `char*`
- 动态数组
- 句柄

那么你不能只创建容器就完事，还要考虑：

- 拷贝时怎么复制资源
- 删除时怎么释放资源

建议主动设置：

```c
XContainerSetDataCopyMethod(obj, my_copy);
XContainerSetDataMoveMethod(obj, my_move);
XContainerSetDataDeinitMethod(obj, my_deinit);
```

这是这套库真正“高级”的地方，也是最容易踩坑的地方。

## 5. 内存分配是可替换的

`XMemory` 做了一层封装，当前版本会根据平台选择底层分配函数。

在你这个工程里，它默认不是纯 PC 版 `malloc/free` 路线，而是更偏向嵌入式内存接口。所以你读源码时要带着这个意识：

- XinYueC 不是把内存策略写死了
- 它是想允许你把库挂到自己的内存管理器上

这对单片机很有价值。

## 6. 当前版本里值得特别注意的地方

### 注意点 1：`XVector_sort_base` 使用前最好先设置 `compare`

从当前实现看，`XVector_sort_base()` 会把容器里的 `m_compare` 传给排序算法。

如果没提前设置，排序行为就不可靠。

### 注意点 2：`XQueue_receive_base` 的实现和注释语义不完全一致

从当前 `XQueue.c` 的实现看，它会把队头数据拷贝到缓冲区，但没有在函数内部真正执行出队。

也就是说，学习时要区分：

- 头文件/注释写了什么
- 源码真实做了什么

### 注意点 3：注释字符编码有些地方已经乱码

你现在仓库里部分注释已经出现编码问题，所以阅读源码时不要过度依赖中文注释，要多看函数签名和实现本身。

## 7. 最适合你的学习顺序

结合你现在已经移植成功这一点，我建议你按这个节奏走：

1. 先把 `XVector<int>` 和原生数组排序跑通
2. 再写一个自定义结构体 + 比较函数
3. 再补 `copy/deinit` 回调，理解资源管理
4. 然后再看 `XStack / XQueue / XPriorityQueue`
5. 最后再回头读 `XClass / XVtable`

这样不容易一上来就被宏和虚表吓住。
