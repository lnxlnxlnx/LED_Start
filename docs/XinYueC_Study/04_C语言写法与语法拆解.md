# 04. C 语言写法与语法拆解

这一份不讲“业务功能”，专门讲这套源码里出现的编程写法。

## 1. `typedef` 函数指针

例如：

```c
typedef int32_t(*XCompare)(const void* lhs, const void* rhs);
typedef void (*XFor_each)(void* pvValue, void* args);
```

这是把“函数类型”当成普通类型来命名。好处是：

- 参数列表更整洁
- 回调接口更统一
- 更适合放进结构体

## 2. `void* + typeSize`：C 里最常见的泛型技巧

比如容器不知道里面装的是 `int` 还是 `struct`，那就：

- 用 `void*` 指向一块原始内存
- 用 `typeSize` 表示每个元素多大

访问第 `i` 个元素时再手动算地址：

```c
(char*)data + i * typeSize
```

这是整套容器设计最基础的写法。

## 3. 强制类型转换

这个库里你会经常看到：

```c
(XVector*)obj
(XContainerObject*)obj
(char*)ptr
```

它们分别常见于：

- 父子类型转换
- 字节级地址偏移
- 回调函数类型匹配

这类写法在底层框架里很常见，但一定要非常清楚对象真实类型。

## 4. 宏包装“伪模板”

例如：

```c
#define XVector_Create(Type) XVector_create(sizeof(Type))
#define XVector_Push_Back_Base(this_vector,type,value){type t=value;XVector_push_back_base(this_vector,&t);}
```

这就是典型的 C 宏做“语法糖”：

- 避免手写 `sizeof(Type)`
- 帮你自动造一个临时变量
- 让接口看起来更像模板容器

缺点也明显：

- 宏没有真正类型检查
- 调试展开后不直观

## 5. 条件编译

你会经常看到：

```c
#if !defined(XVECTOR_H) && XVector_ON
```

这类写法同时做两件事：

- 防止头文件重复包含
- 根据配置开关裁剪模块

在嵌入式库里很常见，因为可以节省 ROM 和 RAM。

## 6. `extern "C"`

```c
#ifdef __cplusplus
extern "C" {
#endif
```

这个不是 C 语法，而是“让 C 头文件兼容 C++ 编译器”的标准写法。

含义是：

- 如果以后用 C++ 工程包含这些头文件
- 不要改函数名链接规则

## 7. 首成员继承

这是本库最核心的一种写法。

```c
typedef struct XPriorityQueue
{
    XVector m_vector;
    XSortOrder m_order;
} XPriorityQueue;
```

本质是在复用内存布局，让“子对象”能被当成“父对象”使用。

这是 C 做对象系统时非常常见的技巧。

## 8. 虚函数表派发

`XClassGetVirtualFunc(...)` 这种宏，核心意思就是：

- 从对象里找到虚函数表
- 按固定槽位取函数指针
- 强转成目标函数类型
- 再调用

这和 C++ 虚函数的思想非常像，只是这里全部手工实现。

## 9. “拷贝 / 移动 / 析构”回调

在普通 C 里，很多人只会写：

```c
memcpy(dst, src, size);
```

但这个库进一步区分了：

- 浅拷贝是否够用
- 是否要转移资源所有权
- 元素销毁时是否要释放内部成员

所以它引入了：

- `XCDataCopyMethod`
- `XCDataMoveMethod`
- `XCDataDeinitMethod`

这实际上是在向 C++ 的“拷贝构造 / 移动构造 / 析构函数”靠近。

## 10. 迭代器宏

例如：

```c
for_each_iterator(container, XVector, it)
{
    ...
}
```

它不是语言自带循环，而是宏展开后的普通 `for`。

这种写法的优点是：

- 用法统一
- 可读性比手写地址偏移好

缺点是：

- 出问题时要有宏展开思维

## 11. 你能从这套源码学到什么

如果你认真读这套库，最值得学的不是“某个排序函数怎么写”，而是这些底层能力：

- 如何在 C 里做模块化抽象
- 如何把通用逻辑和具体实现拆层
- 如何设计可替换的回调接口
- 如何在嵌入式场景里保留一定的工程化结构

这几件事比单个 API 本身更有价值。
