# 09. 补充细节：宏枚举与虚表继承多态

这一份只回答三个问题：

1. “宏枚举”到底是什么
2. 虚函数表“追加操作”到底在做什么
3. 类继承和多态在 C 里是怎么落地的

---

## 1. 先把“宏枚举”翻译成人话

在 `XinYueC/include/XVtable.h` 里有这组宏：

```c
#define XCLASS_DEFINE_BEGING(Class)  enum Class##VtableEnum{
#define XCLASS_DEFINE_ENUM(Class,Value) E##Class##_##Value
#define XCLASS_DEFINE_END(Class)    XCLASS_VTABLE_GET_SIZE(Class)};
#define XCLASS_VTABLE_GET_SIZE(Class)   E##Class##_END_SIZE
```

它们的作用是：自动生成“虚函数槽位编号”。

例如在 `XVector.h` 里写：

```c
XCLASS_DEFINE_BEGING(XVector)
XCLASS_DEFINE_ENUM(XVector, Resize) = XCLASS_VTABLE_GET_SIZE(XContainerObject),
XCLASS_DEFINE_ENUM(XVector, Push_Front),
XCLASS_DEFINE_ENUM(XVector, Push_Back),
...
XCLASS_DEFINE_END(XVector)
```

可理解成普通 `enum`：

```c
enum XVectorVtableEnum {
    EXVector_Resize = EXContainerObject_END_SIZE,
    EXVector_Push_Front,
    EXVector_Push_Back,
    ...
    EXVector_END_SIZE
};
```

所以“宏枚举”不是新语法，只是用宏批量生成枚举常量。

---

## 2. 为什么要用“枚举索引”

因为虚函数表本质是函数指针数组，调用时必须知道“第几个槽位”。

XinYueC 的调用宏：

```c
#define XVtableGetFunc(Vtable,Offset,Type) ((Type)((((XVtable*)Vtable)->data)[Offset]))
#define XClassGetVirtualFunc(Object,Offset,Type) XVtableGetFunc((XClassGetVtable(Object)),Offset,Type)
```

意思是：

- 用 `Offset`（如 `EXVector_Push_Back`）索引虚表
- 取出函数指针
- 强转成目标函数类型并调用

这就是“枚举编号 -> 虚表槽位 -> 函数调用”的链路。

---

## 3. 虚函数表“追加操作”是怎么做的

关键看 `XVector_class_init()`（`XVector_virtual.c`）：

1. 创建或拿到静态虚表
2. 继承父类虚表：`XVTABLE_INHERIT_DEFAULT(XContainerObject_class_init())`
3. 追加子类函数列表：`XVTABLE_ADD_FUNC_LIST_DEFAULT(table)`
4. 必要时重载父类槽位：`XVTABLE_OVERLOAD_DEFAULT(...)`

其中第 3 步“追加”最关键。它把：

```c
void* table[] = {
    VXVector_resize,
    VXVector_push_front, VXVector_push_back,
    ...
    VXVector_sort
};
```

按顺序压到虚表尾部。  
这个顺序必须与 `enum` 里 `EXVector_...` 的顺序严格一致。

### 一句话理解“追加”

父类先占前半段槽位，子类把自己的函数顺序接在后面，这样就实现了“父类接口 + 子类扩展接口”。

### 为什么每次写子类都要先 `XVTABLE_INHERIT_DEFAULT(...)`

这是“继承链不断裂”的关键步骤。

`XVTABLE_INHERIT_DEFAULT(VtableBase)` 宏最终会调用：

```c
XVtable_append_vtable(XVTABLE_DEFAULT, VtableBase)
```

而 `XVtable_append_vtable` 的实现本质是把父类虚表整段拷贝到当前虚表前部。  
这样做之后，子类对象才能继续使用父类已定义的槽位（例如 `EXClass_Copy`、`EXContainerObject_Size`）。

如果你省略这一步，后果通常是：

1. 子类虚表里没有父类槽位
2. 后续 `XClassGetVirtualFunc(..., EX父类_某函数, ...)` 取到错误位置
3. 运行时行为错乱或崩溃

所以规范顺序是固定的：

1. `XVTABLE_CREAT_DEFAULT`
2. `XVTABLE_STACK_INIT_DEFAULT(...)` 或 `XVTABLE_HEAP_INIT_DEFAULT`
3. `XVTABLE_INHERIT_DEFAULT(Parent_class_init())`
4. `XVTABLE_ADD_FUNC_LIST_DEFAULT(table)` 追加子类新槽位
5. `XVTABLE_OVERLOAD_DEFAULT(...)` 覆盖父类槽位

你会在多个类里看到同样模式：

- `XContainerObject_class_init` 先继承 `XClass`
- `XVector_class_init` 先继承 `XContainerObject`
- `XQueue_class_init` 先继承 `XContainerObject`

---

## 4. 重载和追加有什么区别

### 追加（append）

- 用于“新增槽位”
- 不改父类已有槽位
- 典型宏：`XVTABLE_ADD_FUNC_LIST_DEFAULT`

### 重载（overload）

- 用于“替换已有槽位”
- 子类覆盖父类某个行为
- 典型宏：`XVTABLE_OVERLOAD_DEFAULT`

在 `XVector_class_init()` 里：

- `EXContainerObject_Clear` 被重载为 `VXVector_clear`
- `EXClass_Copy/Move` 被重载为 `VXClass_copy/move`

这就是 C 版“override”。

---

## 5. 继承在 C 里怎么实现

核心技巧：首成员继承。

`XVector` 定义：

```c
typedef struct XVector
{
    XContainerObject m_class;
} XVector;
```

因为首成员地址与结构体起始地址一致，`XVector*` 可转为 `XContainerObject*`，再转为 `XClass*`。  
这样子类就能复用父类接口和虚表结构。

---

## 6. 多态在 C 里怎么实现

多态不是编译器魔法，而是运行时派发：

1. 对象里存 `m_vtable`
2. API 用枚举索引取函数指针
3. 调到该对象真实绑定的实现

示例：`XVector_push_back_base()` 并不直接写死 `VXVector_push_back`，而是通过：

```c
XClassGetVirtualFunc(this_vector, EXVector_Push_Back, ...)(...)
```

只要对象 `m_vtable` 指向的是 `XVector` 的虚表，就会调到 `XVector` 实现；换成其他对象就会调到其对应实现。  
这就是动态绑定（多态）。

---

## 7. 用一条真实链路串起来（push_back）

1. `XVector_init` 时把 `m_vtable` 指到 `XVector_class_init()`
2. `XVector_class_init()` 先继承父虚表，再追加 `VXVector_push_back`
3. 调用 `XVector_push_back_base()`
4. 通过 `EXVector_Push_Back` 索引拿到函数指针
5. 最终执行 `VXVector_push_back`

你可以把它记成：

**初始化绑表 -> class_init 填表 -> base API 按索引派发 -> 命中真实实现**

---

## 8. 这套宏最容易踩的坑

1. `enum` 顺序和 `table[]` 顺序不一致，调用会错位  
2. 函数签名强转错误，编译可能过但运行不稳定  
3. 忘记在 `init` 里绑定 `m_vtable`，派发直接失效  
4. 该“重载”的地方只做了“追加”，导致父类行为没被替换

---

## 9. 看宏不迷路的阅读顺序

1. 先看 `XVector.h` 里的 `EXVector_...` 枚举
2. 再看 `XVector_virtual.c` 的 `table[]` 填充顺序
3. 再看 `XVector.c` 的 `XClassGetVirtualFunc(...)` 调用
4. 最后回到 `XClass.h` / `XVtable.h` 看宏定义

按这个顺序，你能把“宏看起来很玄”的部分还原成普通的“数组索引调用”。
