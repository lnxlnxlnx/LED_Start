# 03. 源码设计与分层

这一份是“为什么它会写成这样”。

## 1. 先看总分层

可以把 XinYueC 的核心层次理解成：

1. `XVtable`
2. `XClass`
3. `XContainerObject`
4. 各种具体容器，如 `XVector / XListSLinked / XQueue / XPriorityQueue`

这四层分别解决不同问题。

## 2. `XVtable`：虚函数表本体

`XVtable` 本质上就是一个“函数指针数组”：

- `data`：存函数指针
- `size`：当前有多少槽位
- `capacity`：容量
- `isStack`：是栈上静态表还是堆上动态表

它的作用只有一个：给“多态调用”提供基础设施。

## 3. `XClass`：所有“类”的共同起点

`XClass` 非常小，核心成员只有一个：

```c
typedef struct XClass
{
    XVtable* m_vtable;
} XClass;
```

也就是说，它把“对象”和“虚函数表”绑定起来了。

后续所有派生类型都可以通过它完成：

- 拷贝
- 移动
- 析构

## 4. 这个库怎么在 C 里模拟“继承”

关键技巧是：把父类结构体放在子类结构体的第一个成员。

例如 `XVector`：

```c
typedef struct XVector
{
    XContainerObject m_class;
} XVector;
```

这样做的意义是：

- 子类地址和首成员地址相同
- 可以把 `XVector*` 当成 `XContainerObject*` 或 `XClass*` 使用

这就是典型的“首成员继承”技巧。

## 5. `XContainerObject`：泛型容器基类

这一层最重要，因为它解决了“容器如何装任意类型数据”。

它里面有几个关键字段：

- `m_data`：真正的数据区
- `m_capacity`：容量
- `m_size`：当前元素数
- `m_typeSize`：单个元素大小
- `m_compare`：比较函数
- `m_dataCopyMethod`：拷贝函数
- `m_dataMoveMethod`：移动函数
- `m_dataDeinitMethod`：析构函数

这就是泛型容器的核心套路：

- 不知道元素是什么类型
- 但知道每个元素占多少字节
- 再把“元素怎么比较、怎么拷贝、怎么释放”交给回调函数

## 6. `XVector` 为什么分成 `.c` 和 `_virtual.c`

这是这套代码很值得学的一个习惯。

### `XVector.c`

更像“对外包装层”：

- 参数检查
- 统一入口
- 通过虚函数表转发

### `XVector_virtual.c`

更像“真实实现层”：

- `push_back`
- `insert`
- `clear`
- `sort`
- `find`

这样拆分后，外层 API 比较统一，底层实现也更集中。

## 7. 多态调用是怎么完成的

以 `XVector_push_back_base` 为例，流程大致是：

1. 用户调用统一接口
2. 代码从对象里拿到 `m_vtable`
3. 通过枚举索引找到对应函数指针
4. 调用真实实现

这就是 C 版“虚函数派发”。

## 8. 为什么有一堆枚举和宏

你会看到类似：

```c
XCLASS_DEFINE_BEGING(XVector)
XCLASS_DEFINE_ENUM(XVector, Resize)
XCLASS_DEFINE_ENUM(XVector, Push_Front)
...
```

它的意义不是为了炫技，而是为了做两件事：

1. 给每个虚函数一个稳定索引
2. 让子类在父类虚表基础上继续追加函数槽位

所以这套宏本质上是在“自动管理虚表下标”。

## 9. 派生容器为什么写得很薄

比如 `XStack` 其实大部分接口都是宏映射：

- `push` 映射到 `XVector_push_back`
- `pop` 映射到 `XVector_pop_back`
- `top` 映射到 `XVector_back`

这种写法体现的是“复用优先”：

- 不重复造轮子
- 只换语义，不重写底层逻辑

`XPriorityQueue` 也是类似思路，只是额外加了优先级规则。

## 10. 这套设计的优点

- 在纯 C 环境里实现了相当强的抽象能力
- 一套容器框架可以复用很多底层机制
- 适合做嵌入式可移植组件
- 便于把“内存策略”和“元素行为”注入进去

## 11. 这套设计的代价

- 宏很多，阅读门槛高
- 类型安全弱，很多错误只能运行时发现
- 调试时不如普通 C 结构直观
- 注释、命名、真实行为之间有时不完全一致

## 12. 建议的源码阅读顺序

如果你准备正式啃源码，建议顺序是：

1. `XVector.h`
2. `XVector.c`
3. `XVector_virtual.c`
4. `XContainerObject.h`
5. `XClass.h`
6. `XVtable.h`
7. 再回头看 `XStack.h / XQueue.h / XPriorityQueue.h`

这样最不容易迷路。
