# lw_oopc vs XinYueC 对比（本项目视角）

一句话结论：如果你要“轻量 OOP 骨架”，优先 `lw_oopc`；如果你要“容器 + 算法 + 工程化抽象”，优先 `XinYueC`。

## 🎯 1. 对比总览

| 维度 | lw_oopc | XinYueC |
|---|---|---|
| 定位 | OOP 宏骨架（类/继承/接口/析构） | 通用基础库（容器、算法、比较器、内存抽象） |
| 学习曲线 | 低到中 | 中到高 |
| 代码规模 | 小 | 大 |
| 功能多样性 | 低到中（偏机制） | 高（偏能力） |
| 适用场景 | 业务对象建模、模块解耦 | 数据结构密集、通用组件复用 |
| 典型路径 | `EXTENSION/lw_oopc`、`IO/OOP` | `EXTENSION/XinYueC`、`IO/XinYue` |

## ✨ 2. 功能多样性对比

### 2.1 lw_oopc 强项

1. 用宏快速搭类层次：`CLASS/ABS_CLASS/INTERFACE`
2. 支持父子类和接口组合：`EXTENDS/IMPLEMENTS`
3. 支持显式析构链：`DTOR/END_DTOR`
4. 结构清晰，适合讲清“对象生命周期”

### 2.2 XinYueC 强项

1. 容器丰富：`XVector/XList/XQueue/XStack/...`
2. 算法和工具丰富：`XQuickSort/XCompare/...`
3. 更偏“可复用基础设施”，不是只提供 OOP 语法层
4. 在复杂数据处理任务里更省开发时间

### 2.3 取舍建议

1. 业务逻辑以“设备对象、状态对象”建模为主：选 `lw_oopc`
2. 业务逻辑以“批量数据处理、容器操作”为主：选 `XinYueC`
3. 混用时建议分层：对象模型用 `lw_oopc`，数据结构用 `XinYueC`

## 🔍 3. 实现技巧对比

### 3.1 lw_oopc 实现技巧

1. 用“组合优先于深继承”：减少 `SUPER_PTR_2/3` 依赖
2. 统一对象出口：所有 `new` 对应 `delete`，禁止直接 `free`
3. 在 `dtor/finalize` 里闭合资源释放链，不做跨层偷释放
4. 先接口后实现：先定义 `INTERFACE` 再落地具体类

### 3.2 XinYueC 实现技巧

1. 先用现成容器，不要重复造轮子
2. 比较器策略集中管理，减少排序/查找行为漂移
3. 用模块边界限制模板式宏扩散，避免代码可读性下降
4. 结合 `IO/XinYue` 示例逐步接入，减少一次性改动面

## ⚡ 4. 性能对比（趋势 + 可测方法）

> 说明：下面先给“趋势判断”，再给“你项目内可复现的测量方法”。  
> 不给拍脑袋绝对数值，避免误导调优方向。

### 4.1 趋势判断（STM32F103 这类 MCU）

1. 纯虚调用开销：两者本质都走函数指针，单次调用开销同量级。
2. 代码体积：只引入 OOP 机制时，`lw_oopc` 通常更小。
3. 数据结构密集场景：`XinYueC` 往往更快落地，因为容器与算法现成。
4. 动态分配敏感场景：两者都应控制 `malloc/free` 频率，热点路径尽量对象池化或静态化。

### 4.2 基准建议（可直接在本仓库做）

建议分三组测试，统一在同一编译选项下比较：

1. `对象创建/销毁吞吐`
2. `虚调用吞吐`
3. `典型业务场景总耗时`（如“采样数据入队 + 排序 + 输出”）

建议记录指标：

1. `.text/.data/.bss` 体积（从 map 文件读取）
2. 总耗时（`delay_get_ms()` 或硬件定时器）
3. 峰值内存占用（若可测）

### 4.3 最小测量模板（示意）

```c
u32 t0 = delay_get_ms();
for (int i = 0; i < 10000; i++)
{
    /* 被测操作：new/delete 或 虚函数调用 */
}
u32 t1 = delay_get_ms();
printf("cost=%lu ms\r\n", (unsigned long)(t1 - t0));
```

为减少 1ms 粒度误差，建议：

1. 循环次数至少 1e4 级别
2. 每项跑 5 次取中位数
3. 关闭无关日志后再测

## ✅ 5. 选型建议（第一性原理）

先回答三个问题，再选库：

1. 你现在的核心痛点是“对象建模混乱”还是“容器算法缺失”？
2. 你要优化的是“运行时性能”还是“开发与维护效率”？
3. 未来 3 个月，代码主要增长在“业务对象”还是“通用数据处理”？

对应决策：

1. 以对象建模为主：`lw_oopc`
2. 以通用能力复用为主：`XinYueC`
3. 两者都强需求：分层混用，不要在同一层混杂两套抽象

## 🔥 6. 本项目落地建议

1. 在 `IO/OOP` 继续扩展 `lw_oopc` 业务对象测试（如设备接口多实现）。
2. 在 `IO/XinYue` 保持容器/算法测试独立，避免职责交叉。
3. 新模块提交时标注“用哪套抽象以及理由”，避免后续架构漂移。
