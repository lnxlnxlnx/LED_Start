# 06. 从需求到代码：模式落地模板

这份是“怎么写、怎么用”的操作模板。  
你可以把任何一个模式按这 7 步落地到项目中。

## Step 1. 写痛点（1句话）

模板：

`当前问题是 ______，导致 ______，每次新增 ______ 都要改 ______。`

示例：

`当前问题是传感器驱动接口不统一，导致业务层充满 if/else，每次新增设备都要改业务逻辑。`

## Step 2. 选模式（1个主模式 + 0~1个辅模式）

模板：

- 主模式：解决核心痛点
- 辅模式：只在必要时补充

示例：

- 主模式：Adapter（统一驱动接口）
- 辅模式：Factory Method（按设备类型创建适配器）

## Step 3. 定义稳定接口（先写 .h）

模板：

```c
typedef struct DeviceOps
{
    bool (*init)(void* self);
    int  (*read)(void* self, uint8_t* buf, size_t n);
    void (*deinit)(void* self);
} DeviceOps;
```

原则：

1. 先只放“稳定动作”
2. 参数和返回值语义写清
3. 暂不暴露底层结构细节

## Step 4. 实现一个最小可跑版本

先只支持 1 个设备，把链路打通：

1. `create`
2. `init`
3. `read`
4. `deinit`

然后再扩展第二个设备验证抽象是否成立。

## Step 5. 接入现有模块（小步替换）

替换顺序建议：

1. 先替换调用入口
2. 再替换内部实现
3. 最后删除旧分支

不要一次全量重构，避免定位问题困难。

## Step 6. 做最小验证

每个模式改造都至少做三种验证：

1. 正常路径（功能可用）
2. 异常路径（空指针/错误参数）
3. 扩展路径（新增一个实现后主流程不改）

## Step 7. 写复盘（固定模板）

复盘模板：

1. 这个模式实际解决了什么痛点？
2. 引入它增加了什么复杂度？
3. 下次同类问题还会继续用它吗？为什么？

## 三个常用模式的“可抄作业”骨架

## A. Strategy 骨架

```c
typedef int (*CalcFn)(int a, int b);

typedef struct
{
    CalcFn fn;
} CalcStrategy;

int do_calc(CalcStrategy* s, int a, int b)
{
    if (s == NULL || s->fn == NULL) return 0;
    return s->fn(a, b);
}
```

嵌入式落点：滤波策略、比较策略、编码策略。

## B. State 骨架

```c
typedef enum { ST_IDLE, ST_RUN, ST_ERR } State;
typedef enum { EV_START, EV_STOP, EV_FAULT } Event;

typedef struct
{
    State state;
} Ctx;

void dispatch(Ctx* c, Event e)
{
    switch (c->state) {
    case ST_IDLE:
        if (e == EV_START) c->state = ST_RUN;
        break;
    case ST_RUN:
        if (e == EV_STOP) c->state = ST_IDLE;
        if (e == EV_FAULT) c->state = ST_ERR;
        break;
    case ST_ERR:
        if (e == EV_STOP) c->state = ST_IDLE;
        break;
    }
}
```

嵌入式落点：按键流程、通信流程、设备工作流。

## C. Adapter 骨架

```c
typedef struct
{
    int (*read_temp)(void* self);
} TempOps;

typedef struct
{
    TempOps ops;
    void* impl;
} TempDevice;
```

嵌入式落点：统一不同温度传感器驱动接口。

## 新手常见误区（务必避免）

1. 见到问题先套模式，而不是先写清痛点。
2. 一次引入多个模式，导致复杂度飙升。
3. 只改目录命名，不改依赖结构。
4. 没有验证“新增一个实现是否更容易”。

---

## 一周练习建议（可直接执行）

1. 第一天：选一个痛点，写 Step1~Step3
2. 第二天：实现最小可跑版本
3. 第三天：接入现有模块并保留回退路径
4. 第四天：完成三类验证
5. 第五天：写复盘并提交文档

按这个节奏，你会从“知道模式名”变成“能把模式用于项目”。
