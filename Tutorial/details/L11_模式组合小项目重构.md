# L11 模式组合小项目重构（第 11 周）

## 1. 学习目标

这节课结束后，你应该能做到：

1. 把多个模式组合到同一条真实功能链路中。
2. 从“能跑”升级到“可扩展、可维护”的工程结构。
3. 输出一份重构前后对比说明。

## 2. 知识点讲解（初学者版）

## 2.1 为什么要做模式组合

单个模式只解决一个局部问题。  
真实项目通常是“多个问题叠加”：

- 创建复杂（工厂）
- 算法会变（策略）
- 行为随状态变化（状态）
- 外部接口不一致（适配器）

所以需要组合，而不是孤立学习。

## 2.2 建议的组合链路

本周建议实战链路：

1. 工厂：统一创建设备
2. 适配器：统一读取数据
3. 策略：选择告警判定算法
4. 状态：根据告警等级切换 LED 行为

## 2.3 三问法：什么时候用、映射到哪里、第一步做什么

### 什么时候用

当你要做“一个功能闭环”，并且预计后续会继续迭代时使用组合方案。

### 在本项目映射到哪里

- 设备创建：`USER/main.c` + 工厂模块
- 数据读取：`HARDWARE/*` + 适配层
- 告警判定：`IO/` 策略模块
- 状态执行：`IO/key_led_one.c`

### 第一最小实现步

先做“传感器读取 -> 告警判定 -> LED 状态更新”的最短闭环。

## 3. 本仓库动手步骤

目标：完成一个可演示的小项目重构。

### Step 1：选定业务场景

建议用“温度阈值告警 + LED 提示”：

- 正常：慢闪
- 预警：快闪
- 告警：常亮

### Step 2：先写分层骨架

建议新增：

- `IO/app_monitor.h/.c`（应用流程）
- `IO/device_factory.*`（工厂）
- `IO/sensor_adapter.*`（适配器）
- `IO/alarm_strategy.*`（策略）

### Step 3：接入状态机

把告警等级映射到 `key_led_one` 状态（或新建独立状态机模块）。

### Step 4：加入日志与验证脚本

通过串口打印：

1. 原始采样值
2. 当前策略结果
3. 状态切换结果

### Step 5：输出重构对比

至少对比这三项：

1. 入口函数数量
2. 关键耦合点数量
3. 新增一个策略所需修改文件数

## 4. 常见坑

1. 一开始就追求“完美架构”，导致迟迟没有可运行闭环。
2. 模式划分过细，文件爆炸，反而增加理解成本。
3. 重构后没有行为回归验证，出现功能退化。

## 5. 课后作业与思考题

## 作业

在现有闭环上再新增一个告警策略（例如带迟滞阈值），并记录改动文件。

## 思考题

1. 哪个模式在你这个项目里收益最大？为什么？
2. 如果要交给新同学接手，你会优先补哪份文档？

## 6. 验收标准

- 至少 3 种模式参与同一功能链路。
- 有“重构前后对比”文字记录。
- 功能行为可通过串口日志稳定复现。
