# L08 适配器与外观统一驱动（第 8 周）

## 1. 学习目标

这节课结束后，你应该能做到：

1. 理解适配器模式与外观模式的区别与配合方式。
2. 把“接口风格不一致”的驱动统一成同一调用习惯。
3. 在本仓库实现一个最小“传感器统一读取入口”。

## 2. 知识点讲解（初学者版）

## 2.1 适配器模式做什么

适配器解决的是“老接口不符合你想要的新接口”。

示例：

- 某驱动返回 `int`
- 另一个驱动通过指针出参返回
- 上层希望统一成 `int sensor_read(float* out);`

## 2.2 外观模式做什么

外观模式解决的是“子系统太复杂，上层不想知道细节”。

示例：

- 上层只调用 `env_service_update()`
- 内部再决定读取 DHT11、DS18B20、RTC、并输出日志

## 2.3 三问法：什么时候用、映射到哪里、第一步做什么

### 什么时候用

当你有多个驱动接口风格不同、上层代码被迫写大量分支时使用。

### 在本项目映射到哪里

- `HARDWARE/DHT11/`
- `HARDWARE/DS18B20/`
- `HARDWARE/RTC/`
- `USER/main.c`（当前聚合处）

### 第一最小实现步

先统一两个模块的读接口：温度读取 + 时间读取。

## 3. 本仓库动手步骤

目标：实现“统一读取接口 + 简化调用入口”。

### Step 1：定义统一数据结构

在 `IO/` 下新建：

- `sensor_adapter.h`
- `sensor_adapter.c`

先定义：

```c
typedef struct {
    float temperature;
    float humidity;
    uint8_t valid;
} SensorData;
```

### Step 2：为不同驱动写适配函数

例如：

- `int adapt_ds18b20_read(SensorData* out);`
- `int adapt_dht11_read(SensorData* out);`

让上层不再直接调用原始驱动细节。

### Step 3：再封一层外观

新增：

`int env_service_update(SensorData* out);`

在内部按优先级调用不同适配器并做兜底。

### Step 4：在 `main.c` 使用外观接口

将原有散落调用替换成一条 `env_service_update()`。

### Step 5：串口输出统一结果

统一打印格式，便于后续上位机解析。

## 4. 常见坑

1. 适配器把底层错误吞掉，导致上层无法判断失败原因。
2. 外观函数做了太多业务判断，变成“超级函数”。
3. 统一接口字段定义不完整，后期又频繁破坏兼容性。

## 5. 课后作业与思考题

## 作业

再给 `RTC` 写一个时间适配器，统一到同一外观输出结构。

## 思考题

1. 适配器和工厂有什么边界？它们能否组合使用？
2. 外观层应该放业务逻辑吗？放多少合适？

## 6. 验收标准

- 至少 2 个驱动被统一到同一适配接口。
- 上层调用入口减少为 1 个外观函数。
- 失败路径有可观察日志与返回码。
