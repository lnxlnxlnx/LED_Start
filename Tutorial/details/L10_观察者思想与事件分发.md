# L10 观察者思想与事件分发（第 10 周）

## 1. 学习目标

这节课结束后，你应该能做到：

1. 理解观察者思想（发布-订阅）如何降低模块耦合。
2. 在 C 项目中实现一个最小事件总线。
3. 把“单点回调”升级成“多订阅者分发”。

## 2. 知识点讲解（初学者版）

## 2.1 什么是观察者思想

核心关系：

- 发布者（产生事件）
- 订阅者（处理事件）
- 总线/中心（负责分发）

发布者不需要知道谁在监听，这就是解耦价值。

## 2.2 在嵌入式中的典型用途

- 按键事件触发多个动作（LED、蜂鸣器、串口提示）
- 传感器阈值触发告警与记录
- 通讯模块状态变化通知多个子系统

## 2.3 三问法：什么时候用、映射到哪里、第一步做什么

### 什么时候用

当同一事件需要被多个模块响应，且响应逻辑会持续增加时使用。

### 在本项目映射到哪里

- 事件源：`HARDWARE/EXTI/exti.c`
- 响应方：`IO/key_led_one.c`、`HARDWARE/BEEP/`、`SYSTEM/usart/`

### 第一最小实现步

先做“一个事件 + 两个订阅者”的静态分发版本。

## 3. 本仓库动手步骤

目标：实现 `event_bus` 并把按键事件接入。

### Step 1：定义事件与回调签名

在 `IO/` 新建：

- `event_bus.h`
- `event_bus.c`

定义：

```c
typedef enum { EVT_KEY0_PRESS, EVT_KEY1_PRESS, EVT_KEY_UP } EventType;
typedef void (*EventHandler)(EventType evt, void* data);
```

### Step 2：实现注册与发布

提供最小 API：

1. `event_subscribe(...)`
2. `event_unsubscribe(...)`
3. `event_publish(...)`

### Step 3：接入中断事件

在 `exti.c` 中断里发布事件（或先入队再发布），避免直接耦合业务逻辑。

### Step 4：添加两个订阅者

例如：

1. LED 订阅者：切换灯态
2. UART 订阅者：打印事件日志

### Step 5：验证“新增订阅者不改发布者”

再加一个蜂鸣器订阅者，确认 `exti.c` 无需改动。

## 4. 常见坑

1. 在订阅回调中做阻塞操作，拖慢整个分发链。
2. 订阅者列表边遍历边修改，导致迭代失效。
3. 忘记取消订阅，留下悬空回调。

## 5. 课后作业与思考题

## 作业

为事件总线加“事件计数器”，统计每类事件触发次数。

## 思考题

1. 观察者和命令队列的边界在哪里？
2. 什么时候事件分发应该同步执行，什么时候应异步？

## 6. 验收标准

- 至少实现 2 个订阅者响应同一事件。
- 发布者代码不直接依赖订阅者实现。
- 新增订阅者时，发布路径不变。
