# L06 工厂模式与统一创建入口（第 6 周）

## 1. 学习目标

这节课结束后，你应该能做到：

1. 理解工厂模式的核心是“把创建逻辑集中管理”。
2. 把分散在各处的 `Init/Create` 流程收敛成单一入口。
3. 在本项目中完成一个最小“设备工厂”原型。

## 2. 知识点讲解（初学者版）

## 2.1 为什么需要工厂模式

在嵌入式项目里，设备初始化常常分散在多个文件中：

- `LED_Init()`
- `KEY_Init()`
- `RTC_Init()`
- `Dac_Init()`

如果调用顺序、条件分支到处都是，后期维护很痛苦。

## 2.2 工厂模式在 C 里的写法

常见做法：

1. 定义“设备类型枚举”
2. 建立“类型 -> 创建函数”映射
3. 对外暴露统一入口 `device_create(type)`

## 2.3 三问法：什么时候用、映射到哪里、第一步做什么

### 什么时候用

当对象创建过程复杂、且对象种类会持续增加时使用。

### 在本项目映射到哪里

- `HARDWARE/LED/led.c`
- `HARDWARE/KEY/key.c`
- `HARDWARE/RTC/rtc.c`
- `USER/main.c`（当前聚合调用入口）

### 第一最小实现步

先把 `LED/KEY/RTC` 三个初始化动作收口到一个 `board_devices_init()`。

## 3. 本仓库动手步骤

目标：实现“单入口初始化 + 可扩展设备表”。

### Step 1：建立工厂头文件

在 `IO/` 下新增：

- `device_factory.h`
- `device_factory.c`

头文件先定义：

```c
typedef enum {
    DEV_LED,
    DEV_KEY,
    DEV_RTC
} DeviceType;

int device_create(DeviceType type);
```

### Step 2：实现类型分发

在 `device_factory.c` 用 `switch` 或函数表把 `DeviceType` 分发到具体初始化函数。

### Step 3：把 `main.c` 接入工厂入口

将 `USER/main.c` 中分散的单设备初始化，逐步替换为工厂调用。

### Step 4：加日志与错误码

配合 `easylogger`，在创建成功/失败时打印统一格式日志，便于调试。

### Step 5：验证扩展性

新增一个类型占位（例如 `DEV_DHT11`），验证扩展时不需要改调用方结构。

## 4. 常见坑

1. 工厂只做了函数包装，但没有统一错误处理。
2. 创建设备时隐式依赖调用顺序（例如先时钟后外设）却没写清规则。
3. 设备增多后 `switch` 过长，没有抽象成表驱动。

## 5. 课后作业与思考题

## 作业

把 `LED/KEY/RTC` 接口接入工厂，写一段“初始化日志总览”输出。

## 思考题

1. 工厂模式和“直接在 main 里写初始化”相比，维护成本差异在哪？
2. 什么时候要从“简单工厂”升级为“抽象工厂”？

## 6. 验收标准

- `main.c` 至少有 3 个初始化动作迁移到统一工厂入口。
- 工厂返回值可用于判断成功/失败。
- 新增设备类型时，调用方基本不改结构。
