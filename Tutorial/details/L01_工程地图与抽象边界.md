# L01 工程地图与抽象边界（第 1 周）

## 1. 学习目标

这节课结束后，你应该能做到：

1. 看懂当前仓库的主要目录职责。
2. 说清楚“模块边界”是什么。
3. 为一个模块写出最小接口清单（输入/输出/状态）。

## 2. 知识点讲解（初学者版）

## 2.1 什么是“工程地图”

工程地图 = “哪个目录负责什么”。

你当前仓库可以先这样理解：

- `HARDWARE/`：外设驱动实现层（LED、KEY、RTC 等）
- `SYSTEM/`：系统基础能力（串口、延时、系统初始化）
- `USER/`：应用入口（`main.c`、中断、工程配置）
- `IO/`：功能组合/测试入口
- `XinYueC/`：通用容器与算法库

如果没有工程地图，后续学模式会变成“知道名词，不知道落在哪里”。

## 2.2 什么是“抽象边界”

抽象边界 = “模块对外只暴露必要接口，不暴露内部细节”。

错误示例：

- 业务层直接操作底层寄存器细节。
- 上层到处调用某个驱动的内部函数。

正确方向：

- 上层只依赖 `.h` 的稳定接口。
- 实现细节留在 `.c` 文件里。

## 2.3 三个最基础的边界规则

1. 单向依赖：上层可以调下层，下层不要反向依赖上层。
2. 一处入口：每个模块应有统一初始化/调用入口。
3. 状态可见：模块必须明确自己的状态（未初始化、就绪、错误）。

## 3. 本仓库动手步骤

目标：为 `HARDWARE/LED` 做一页“模块接口清单”。

### Step 1：看接口

打开 `HARDWARE/LED/led.h`，记录公开函数。

### Step 2：看实现

打开 `HARDWARE/LED/led.c`，记录每个函数内部依赖了谁（GPIO、delay、宏等）。

### Step 3：写接口清单（建议模板）

```text
模块名：LED
输入：初始化参数/控制命令
输出：硬件状态变化（亮灭）
状态：未初始化、已初始化
公开接口：LED_Init, LED0/LED1 控制宏或函数
依赖：GPIO、时钟、可能的delay
```

### Step 4：画依赖方向

用 5-10 行文字画出：

`USER/main.c -> 功能逻辑 -> HARDWARE/LED -> STM32F10x_FWLib`

## 4. 常见坑

1. 把“目录位置”当成“抽象边界”，但接口仍然耦合。
2. 只看 `.h` 不看 `.c`，导致不清楚真实依赖。
3. 初始化函数分散在多个地方，调用顺序混乱。

## 5. 课后作业与思考题

## 作业

再选一个模块（`HARDWARE/KEY` 或 `SYSTEM/usart`），按同样模板写接口清单。

## 思考题

1. 如果以后要把 LED 驱动替换成另一块板卡，当前接口会不会改？
2. 如果会改，说明你当前边界设计哪里不稳定？
