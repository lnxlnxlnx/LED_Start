# L04 策略模式与可替换算法（第 4 周）

## 1. 学习目标

这节课结束后，你应该能做到：

1. 理解“策略模式”在 C 语言里本质是“函数指针 + 统一调用约定”。
2. 能把“固定写死算法”改成“运行时可切换算法”。
3. 在本仓库里完成一次排序/比较策略切换演示。

## 2. 知识点讲解（初学者版）

## 2.1 什么是策略模式

一句话：把“会变的行为”单独抽出来，通过接口注入，不写死在业务流程里。

在 C 里，最常见形态是：

- 用函数指针表示“策略函数”
- 业务代码只认统一函数签名
- 具体策略由外部决定

## 2.2 为什么嵌入式项目需要它

常见场景：

- 同一组数据，有时升序，有时降序
- 同一传感器数据，要支持不同滤波规则（均值/中值/阈值）
- 不同板卡或业务模式下，算法选择不同

如果不做策略抽象，最后会充满 `if/else`，修改风险很高。

## 2.3 三问法：什么时候用、映射到哪里、第一步做什么

### 什么时候用

当你发现“流程不变，但某一步算法经常变”时就该用。

### 在本项目映射到哪里

- `XinYueC` 的排序与比较天然适合策略化。
- 可重点看：`XinYueC/include/XSort.h`、`XinYueC/include/XCompare.h`、`IO/XinYue/test_quickSort.h`。

### 第一最小实现步

先实现两种比较策略（升序/降序），并通过同一个排序入口切换运行。

## 3. 本仓库动手步骤

目标：在不改排序主流程的前提下，切换比较策略并打印结果。

### Step 1：读现有测试入口

查看：

- `IO/XinYue/test_quickSort.h`
- `USER/main.c`

确认当前 `test_arr_quicksort()` 与 `test_vector_quicksort()` 的调用路径。

### Step 2：补两套比较策略

在 `IO/XinYue/test_quickSort.h` 增加：

1. `Compare_Int_Asc`
2. `Compare_Int_Desc`

要求签名一致，便于统一注入。

### Step 3：统一策略入口

新增一个策略选择函数，例如：

```c
typedef int (*CompareFn)(const void*, const void*);
CompareFn select_compare_strategy(uint8_t mode);
```

`mode=0` 返回升序，`mode=1` 返回降序。

### Step 4：接入排序流程

在排序调用处，把固定比较函数替换成 `select_compare_strategy(mode)`。

### Step 5：串口验证

在 `USER/main.c` 中先后调用两种模式，串口打印排序前后数组，确认结果确实变化。

## 4. 常见坑

1. 比较函数返回值不规范（不是 `<0/0/>0`），导致排序结果异常。
2. 策略函数签名不一致，强转后运行崩溃。
3. 策略选择和业务状态耦合太深，导致后续很难扩展第三种算法。

## 5. 课后作业与思考题

1. 什么时候不需要策略模式，直接 `if/else` 更合适？

## 6. 验收标准

- 代码中存在至少 2 个可切换策略，且主流程无重复复制。
- 串口日志能展示同一组输入在两种策略下的不同结果。
- 你能清楚说明“变化点”和“稳定点”分别在哪里。
